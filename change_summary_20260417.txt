SMARTINVEN BACKEND - CHANGE SUMMARY
====================================
Date: April 17, 2026
Developer: Sijan Chhetri


OVERVIEW
--------
Implemented a subscription payment verification system with trial period support.
Users can register with a 7-day trial or paid plan. Paid users must upload payment
proof which admins verify before granting access.

CHANGE 1: DATABASE SCHEMA - Added PaymentProof Table
═══════════════════════════════════════════════════════════════════════════════

BEFORE:
-------
- No payment proof tracking existed
- Owner status field existed but no verification workflow

AFTER:
------
New table: payment_proofs

Columns:
- id              : UUID (primary key)
- owner_id        : UUID (foreign key to owners table)
- image_url       : TEXT (S3 key path to payment screenshot)
- status          : TEXT (pending | approved | rejected)
- note            : TEXT (admin rejection reason, nullable)
- reviewed_at     : TIMESTAMP (when admin reviewed, nullable)
- created_at      : TIMESTAMP (when proof was uploaded)

Indexes:
- owner_id (for fast lookup by owner)
- status (for admin filtering pending proofs)

Relation added to Owner model:
- paymentProofs   : PaymentProof[]

WHY:
----
To store payment screenshots uploaded by users and track admin verification status.
Each owner can upload multiple proofs (e.g., if first one is rejected).

MIGRATION:
----------
File: prisma/migrations/20260417145752_add_payment_proof/migration.sql

SQL:
CREATE TABLE "payment_proofs" (
    "id" TEXT NOT NULL,
    "owner_id" TEXT NOT NULL,
    "image_url" TEXT NOT NULL,
    "status" TEXT NOT NULL DEFAULT 'pending',
    "note" TEXT,
    "reviewed_at" TIMESTAMP(3),
    "created_at" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT "payment_proofs_pkey" PRIMARY KEY ("id")
);

CREATE INDEX "payment_proofs_owner_id_idx" ON "payment_proofs"("owner_id");
CREATE INDEX "payment_proofs_status_idx" ON "payment_proofs"("status");

ALTER TABLE "payment_proofs" ADD CONSTRAINT "payment_proofs_owner_id_fkey" 
FOREIGN KEY ("owner_id") REFERENCES "owners"("owner_id") ON DELETE CASCADE ON UPDATE CASCADE;

CHANGE 2: LOGIN LOGIC - Trial Expiry Check
═══════════════════════════════════════════════════════════════════════════════

FILE: src/controllers/authController.js

BEFORE:
-------
- Login only checked if status === "inactive" and blocked with generic message
- No trial period validation
- No differentiation between trial expired and payment pending

AFTER:
------
Added trial expiry logic:

if (owner.status === "trial") {
  const daysSinceCreation = Math.floor(
    (Date.now() - new Date(owner.created_at).getTime()) / (1000 * 60 * 60 * 24)
  );
  if (daysSinceCreation > 7) {
    return sendError(res, 403, "TRIAL_EXPIRED", 
      "Your 7-day trial has expired. Please subscribe to continue.", {
      owner: { ...owner details... }
    });
  }
}

if (owner.status === "inactive") {
  return sendError(res, 403, "ACCOUNT_INACTIVE", 
    "Your payment is pending verification. Please wait for approval.");
}

WHY:
----
- Trial users get 7 days free access calculated from created_at
- After 7 days, login returns TRIAL_EXPIRED with owner info (no token)
- Inactive users (paid but not verified) get clear message about pending verification
- Frontend can detect error_code and show appropriate UI (subscription page vs waiting message)

RESPONSE EXAMPLES:

Trial Expired:
{
  "success": false,
  "error_code": "TRIAL_EXPIRED",
  "message": "Your 7-day trial has expired. Please subscribe to continue.",
  "owner": {
    "owner_id": "...",
    "full_name": "...",
    "email": "...",
    "status": "trial",
    ...
  }
}

Payment Pending:
{
  "success": false,
  "error_code": "ACCOUNT_INACTIVE",
  "message": "Your payment is pending verification. Please wait for approval."
}

CHANGE 3: NEW API - Payment Proof Upload
═══════════════════════════════════════════════════════════════════════════════

FILES CREATED:
--------------
- src/controllers/paymentProofController.js
- src/routes/paymentProofRoutes.js

ENDPOINTS ADDED:
----------------

1. POST /api/payment-proof
   Purpose: Owner uploads payment screenshot
   Auth: None (accepts owner_id as form field)
   Body: multipart/form-data
     - owner_id (text) - from local storage
     - image (file) - payment screenshot (max 5MB)
   Response: { success: true, data: { id, owner_id, image_url, status, created_at } }

2. GET /api/payment-proof/my
   Purpose: Owner views their submitted proofs
   Auth: Required (Bearer token)
   Response: Array of payment proofs

3. GET /api/payment-proof/admin?status=pending
   Purpose: Admin lists proofs by status
   Auth: None (should add admin middleware later)
   Query: status (pending | approved | rejected)
   Response: Array of proofs with owner details

4. PATCH /api/payment-proof/admin/:id/approve
   Purpose: Admin approves payment proof
   Auth: None (should add admin middleware later)
   Effect: Sets proof.status = "approved" AND owner.status = "active"
   Response: { success: true, message: "Payment approved. Owner is now active." }

5. PATCH /api/payment-proof/admin/:id/reject
   Purpose: Admin rejects payment proof
   Auth: None (should add admin middleware later)
   Body: { "note": "reason for rejection" }
   Effect: Sets proof.status = "rejected" with admin note
   Response: { success: true, message: "Payment proof rejected." }

WHY:
----
- Trial expired users have no valid token, so upload endpoint accepts owner_id as form field
- Images stored in S3 under payment-proofs/<owner_id>/<uuid>.png
- Admin can review all pending proofs and approve/reject
- Approval automatically activates the owner account
- Rejection allows admin to provide reason for user to fix and resubmit

MOUNTED IN: src/app.js
app.use("/api/payment-proof", paymentProofRoutes);

USER WORKFLOW SUMMARY
═══════════════════════════════════════════════════════════════════════════════

REGISTRATION:
-------------
1. User registers via POST /api/auth/register
   - Can optionally pass "status": "trial" (default) or "status": "inactive"
   - Trial: Gets 7 days free access immediately
   - Inactive: Must upload payment proof before access

TRIAL USER (7 days):
---------------------
1. Login works normally for 7 days
2. After 7 days, login returns TRIAL_EXPIRED with owner info (no token)
3. Frontend shows subscription page
4. User views payment QR: GET /api/payment-qr/active (no auth needed)
5. User makes payment and uploads screenshot: POST /api/payment-proof
   - Sends owner_id (from local storage) + image file
6. User waits for admin approval
7. After approval, status changes to "active" and login works

PAID USER (inactive → active):
-------------------------------
1. User registers with status="inactive" or selects paid plan
2. Login blocked with ACCOUNT_INACTIVE message
3. User views payment QR: GET /api/payment-qr/active
4. User makes payment and uploads screenshot: POST /api/payment-proof
5. Admin reviews: GET /api/payment-proof/admin?status=pending
6. Admin approves: PATCH /api/payment-proof/admin/:id/approve
7. Owner status changes to "active"
8. User can now login successfully

ADMIN WORKFLOW:
---------------
1. View pending proofs: GET /api/payment-proof/admin?status=pending
2. Review each proof (image_url contains S3 key)
3. Approve: PATCH /api/payment-proof/admin/:id/approve
   OR
   Reject: PATCH /api/payment-proof/admin/:id/reject with note
4. Owner receives access or resubmits based on decision

TECHNICAL NOTES
═══════════════════════════════════════════════════════════════════════════════

TRIAL CALCULATION:
------------------
- Uses owner.created_at timestamp
- No additional column needed
- Calculated on every login: (current_date - created_at) > 7 days
- Efficient and doesn't require cron jobs or scheduled tasks

S3 UPLOAD:
----------
- Uses existing uploadToS3 utility from src/utils/s3.js
- Stores only S3 key in database (not full URL)
- Full URL can be constructed when needed: 
  https://s3-np1.datahub.com.np/smartinevn/<key>

MULTER CONFIGURATION:
---------------------
- Memory storage (no disk writes)
- 5MB file size limit
- No file type restriction (removed to support various screenshot formats)

SECURITY CONSIDERATIONS:
------------------------
- Admin endpoints currently have no auth (TODO: add admin middleware)
- Payment proof upload accepts owner_id without verification (acceptable since
  it only creates a pending record, doesn't grant access)
- Approval is atomic transaction (proof + owner status updated together)
